# M4.1 — v2 tagging (entities split + scope-aware vocabulary)

**Status:** ✅ shipped 2026-05-16. **v3 prompt iteration** shipped 2026-05-17. **v4 prompt iteration** shipped 2026-05-17 (see end of doc); tagger version is now 4.

**Scope:** prompt-level + small trait/storage/drainer changes on top of M4's thoughts-only tagging substrate. No schema migration. Tagger version bump 1→2; operator backfills with `engram tag --rerun --since 1970-01-01T00:00:00Z`.

**What this is not:** selective relations / closed-vocabulary edges between thoughts. That's the M5 candidate. M4.1 is the smaller iteration that lets us validate corpus-level tag coherence and the entities/topics distinction independently first.

## Motivation

The M4 dogfood produced two clear findings on the v1 tagger that this iteration addresses.

### 1. The "memory-systems" finding (category inference)

A thought titled "phrase-driven topics" got `topics: ["tagging", "concept-mapping", "memory-systems"]` even though neither thought contained the word "memory" except in a quoted reference. A refinement of that thought (`137dba1d`, citing the original `6d2ef58e`) got the same three tags despite different prose.

Interpretation: the v1 tagger half-distinguishes proper-noun-style identifiers (lexemes present in the prose) from category labels (inferred from the model's pretrained world model when the prose triangulates to a well-known concept). The v1 schema collapsed both into `topics`, masking the distinction.

Refined model from the dogfood notes: *"topics is phrase-dominant for ~2 of 3 slots; one slot may be domain-category-inferred when the prose triangulates to a well-known concept; reliability of the inferred slot scales with how well-represented the category is in the model's pretraining."*

### 2. Probe 2 — vocabulary divergence at corpus level

When the same author wrote about the same subject in deliberately disjoint prose, the v1 tagger produced divergent topic terms — phrase-driven extraction doesn't cluster across paraphrases. This undermined `tag_filter` utility at corpus level: filtering by `{"topics": ["rust"]}` would miss thoughts where the author wrote "Rust" or used different surface forms.

The natural fix is to feed the tagger existing scope vocabulary as a "controlled vocabulary" hint, so it prefers established terms when they fit and coins new ones only for genuinely unseen concepts.

## Architectural target

### Tags schema (additive, no migration)

`Tags` gains a separate `entities` field alongside the existing `people` and `topics`. Field semantics:

| Field | Semantics |
|---|---|
| `people` | bare names of people (unchanged) |
| `entities` | non-person proper-noun-style identifiers explicitly mentioned: projects, products, libraries, tools, named concepts. **New in M4.1.** |
| `action_items` | imperative phrases (unchanged) |
| `topics` | 1-3 short tag-like subject categories the thought falls under (refined: now explicitly *categories* rather than mixed) |
| `dates_mentioned` | (unchanged) |
| `kind` | closed enum: `observation | task | idea | reference | person_note | session | null` (unchanged) |

Existing v1-tagged rows decode with `entities: vec![]` (the field uses `#[serde(default)]`).

### Tagger trait

Extended with an optional vocabulary context:

```rust
async fn tag(&self, content: &str, vocab: Option<&ScopeVocab>) -> Result<Tags, TaggerError>;

pub struct ScopeVocab {
    pub topics: Vec<String>,
    pub entities: Vec<String>,
}
```

`OpenAICompatibleTagger` renders the vocab into a "controlled vocabulary" section appended to the system prompt. `FakeTagger` records it (so drainer tests can verify) but ignores it for output.

### Storage helper

New `engram_storage::fetch_scope_vocab(pool, scope: &str, limit: i64) -> Result<ScopeVocab, StorageError>`:

```sql
SELECT term AS "term!"
FROM thoughts,
     LATERAL jsonb_array_elements_text(tags->'topics') AS term
WHERE scope = $1 AND retracted_at IS NULL
GROUP BY term
ORDER BY COUNT(*) DESC, term ASC
LIMIT $2
```

…and the matching query for `entities`. Retracted thoughts are excluded so retracted-vocab doesn't bleed back into new captures.

### Drainer wiring

`process_tag_job` in `crates/engram-mcp/src/drain.rs`:

1. Fetches the thought.
2. If `scope_vocab_limit` is `Some(n) > 0`, calls `fetch_scope_vocab(pool, &thought.scope.as_str(), n)`. Empty results normalize to `None` (no point sending an empty vocab section).
3. Calls `tagger.tag(&thought.content, vocab.as_ref())`.

Vocab-fetch failure folds into the transient bucket (job stays queued, attempts++).

### Config

`TaggerConfig` (in `engram-cli`) gains:

- `scope_vocab_enabled: bool` (default `true`).
- `scope_vocab_size: u32` (default `50`).
- Bumped default `model_version: i32 = 2`.

### MCP wire surface

- `SearchHit.tags` already serializes the full `Tags` struct; `entities` propagates for free.
- `SearchThoughtsArgs.tag_filter` `#[schemars(description = ...)]` updated to advertise the new shape and document the entities-vs-topics distinction.
- `SERVER_INSTRUCTIONS` const updated to match. The `server_instructions_advertise_tags_shape_and_kind_enum` regression test pins the entities documentation.

### Operator backfill

```bash
engram tag --rerun --since 1970-01-01T00:00:00Z
```

`find_untagged_or_stale_thoughts(target_tagger_version=2, rerun=true)` walks every thought tagged below v2. The drainer (or one-shot `engram tag` run) re-tags them under v2.

## Risks

- **`tag_filter: {"topics": [...]}` semantics shift on backfill.** Agents that hardcoded queries against v1's `topics` shape may start missing thoughts where v2 reclassifies a term into `entities`. Operator-accepted (single-operator deployment). Documented in `DEVELOPMENT.md`.
- **Vocab ossification.** Early thoughts in a fresh scope disproportionately shape the vocabulary. Mitigation: `scope_vocab_size = 50` is small enough that new terms surface naturally; the prompt instructs the model to coin new terms when prose introduces unseen concepts. Watch during dogfood.
- **Vocab fetch cost.** One JSONB-aggregation query per tag job. Cheap at single-operator corpus scale (dozens-thousands of thoughts). Revisit if the corpus grows past 100K thoughts.
- **Two changes landing together.** If v2 dogfood reveals weirdness, slightly harder to attribute (entities split vs vocab injection). The operator can toggle `scope_vocab_enabled = false` at runtime to isolate the entities-split effect without bumping model_version.

## Dogfood plan (post-merge)

1. `engram tag --rerun --since 1970-01-01T00:00:00Z` against the current corpus.
2. Spot-check entities/topics split on a handful of thoughts. Compare to v1's `topics` distribution on the same thoughts.
3. Re-run Probes 1-3 from the M4 dogfood backlog on v2 outputs.
4. Capture findings as thoughts (matching the established dogfood pattern; cite by `thought_id` in the prose).
5. Decide whether v2 is the stable baseline before opening M5 (selective relations) planning. Open question: does scope vocab injection cause vocabulary ossification within the first ~100 captures, and is the tradeoff net-positive for retrieval?

## Out of scope (deliberate)

- Selective relations / graph edges → M5 (the "Path B-OB1 + edges" candidate).
- Schema migrations → tags is JSONB; the `entities` key is additive.
- A/B between v1 and v2 prompts side-by-side → version bump replaces v1; v1 prompt drops out of code.
- New MCP tools → no surface additions; v2 is internal-shape only.
- Probes 1-3 themselves → dogfood activities, not implementation work.

## Decision log

- **Bundle 1+2 in one iteration, not two.** Both are prompt-shape changes that benefit from landing together (one re-tag pass, one provenance bump). The attribution-difficulty risk if v2 dogfood reveals weirdness is mitigated by the runtime `scope_vocab_enabled` toggle.
- **`entities` is a separate field, not a typed union with `people`.** Keeps `people` queries working (`tag_filter: {"people": [...]}` unchanged), matches the model's existing tendency to handle the two slots distinctly, and avoids a more invasive schema.
- **Vocab as trait parameter, not stored on the tagger.** Per-call vocab supports multi-scope corpora cleanly. The trait's `Option<&ScopeVocab>` lets `FakeTagger` ignore it and lets `OpenAICompatibleTagger` decide rendering.
- **Empty vocab normalizes to `None` in the drainer.** Avoids sending an empty controlled-vocabulary section that would prompt the model to interpret "no established terms" — better to omit the section entirely so the model falls back to default coinage behavior.
- **`scope_vocab_size` default = 50.** Small enough to keep prompt-token cost modest, large enough to cover the active vocabulary in a single-operator corpus. Tunable via TOML; the dogfood phase will validate.

## v3 prompt iteration (2026-05-17)

Dogfood on the v2 tagger surfaced two failure modes. v3 is a prompt-only iteration that addresses both without touching the schema, trait, drainer, or storage. `BUNDLED_TAGGER_VERSION` bumps 2→3; operator runs `engram tag --rerun --since 1970-01-01T00:00:00Z` to backfill.

### Failure mode 1: entities padding

v2 entities included items like `["agent memory protocol", "embedding-based", "lexical signals", "cross-encoder"]` on thoughts that contained no proper nouns. The model treated the required `entities` slot as "fill with whatever looks named-enough in the prose" rather than "empty array if no named entities present." Root cause: the v2 prompt's "named concepts" phrasing in the entities description was the door the model walked through. `tag_filter: {"entities": ["Engram"]}` is rock-solid; `tag_filter: {"entities": ["agent memory protocol"]}` is brittle.

### Failure mode 2: kind drift

Two thoughts changed `kind` between v1 and v2 reruns (8a533e15: task→observation; 047d0ce8: reference→observation). Two probes shipped to diagnose:

**`kind_stability_diagnostic`** (live qwen3-coder:30b on Ollama, N=10 per fixture, vocab=None):

| short_id | stored kind | diagnostic kind (N=10) |
|---|---|---|
| `8a533e15` | observation | task: 10 |
| `047d0ce8` | observation | observation: 10 |
| `22bccb3a` | reference | observation: 10 |
| `5aacd2d8` | reference | reference: 10 |
| `b67db532` | person_note | person_note: 10 |
| `86c3392f` | observation | observation: 10 |

**`kind_stability_diagnostic_with_vocab`** (same fixtures, vocab=Some(<frozen scope vocab>)):

| short_id | scope | stored | vocab-on (N=10) |
|---|---|---|---|
| `8a533e15` | engram.m3.dogfood | observation | observation: 8, task: 2 |
| `047d0ce8` | engram.m3.dogfood | observation | observation: 10 |
| `22bccb3a` | engram.test | reference | observation: 10 |
| `5aacd2d8` | engram.m3.dogfood | reference | reference: 10 |
| `b67db532` | work.tcgplayer | person_note | person_note: 10 |
| `86c3392f` | engram.test | observation | observation: 10 |

Three findings:
1. Within-tagger kind is deterministic at temperature 0.2 — every fixture is 10/10 same kind across reruns at a given vocab state. Falsifies the "kind is just noisy at temp=0.2" hypothesis.
2. Vocab is a real but partial lever on kind: 8a533e15 shifts task→observation at 80% when vocab is supplied. Most other fixtures are stable regardless of vocab.
3. At least one fixture (22bccb3a) has a stored kind that's unreachable from current model+vocab state: both diagnostics produce `observation` 10/10, but the stored value is `reference`. The kind classification drifts across model+vocab+corpus snapshots, not just within a single re-tag.

### v3 prompt changes

Three changes to `BUNDLED_TAGGER_PROMPT`:

1. **Entities tightening.** Replaced "named, proper-noun-style identifiers ... named concepts" with "canonical proper names of specific named things ... recognizable as named entities to someone outside the conversation." Added two organizational/multi-word examples ("TCGplayer", "Cap'n Proto") alongside the technical ones.

2. **Entities anti-padding rule.** Added a capitalized clause: `RETURN [] IF THE THOUGHT CONTAINS NO SUCH NAMED ENTITIES`. Followed by a verbatim list of v2-padding negative examples from the dogfood: "agent memory protocol", "cross-encoder", "embedding-based", "lexical signals", "vector search", "rollout coordinator", "serialization format". The Rules section gets a matching rule.

3. **Kind isolation clause.** Two pieces. The kind field description leads with "intrinsic shape — not its subject area, and not influenced by what other thoughts in this scope tend to be classified as" and gives an inline form→category mapping (factual claim → observation; imperative → task; …). The Rules section adds: "Kind is classified from the thought's intrinsic shape, not from the scope's typical content or any controlled-vocabulary hints below. The vocabulary section, when present, informs topic and entity term choice only."

Also: `reference` category description gets "or a definition of a named thing" appended — small expansion motivated by the 22bccb3a finding, though the diagnostic suggests prompt language alone won't fully address its drift.

### v3 expected behavior (hypotheses to verify post-deploy)

- Entities arrays are shorter on average and empty more often; contents are canonical names only.
- 8a533e15 classifies as `task` more consistently regardless of vocab presence (kind isolation works).
- 22bccb3a may classify as `reference` more often, but the diagnostic suggests it'll probably stay `observation` — the underlying mechanism isn't fully addressable by prompt language.
- No regression on the stably-classified controls.

### v3 dogfood plan

1. Bump local TOML `model_version = 3` (or remove the line so the bundled default applies).
2. `engram tag --rerun --since 1970-01-01T00:00:00Z` against the current corpus.
3. Re-run `kind_stability_diagnostic` and `kind_stability_diagnostic_with_vocab` post-rerun; compare to v2's distributions documented above.
4. Spot-check entities arrays across the corpus — they should be shorter and contain only canonical names.
5. Capture findings as thoughts in `engram.m3.dogfood`.

### v3 acknowledged limits

- The kind-isolation clause is a soft instruction, not a structural fix. The diagnostic suggests it should move bistable cases (8a533e15) but won't fix cross-snapshot drift (22bccb3a).
- Kind classification is now documented as advisory across re-tag cycles. Operators who hardcode `tag_filter: {"kind": "reference"}` should expect a small fraction of relevant thoughts to land under `observation`.
- Selective relations remain the M5 candidate.

## v4 prompt iteration (2026-05-17)

Dogfood on the v3 tagger (full corpus re-tagged, 23 thoughts) revealed v3's entities anti-padding fix was only half-landed and surfaced a new precision concern with scope-aware vocab. v4 is a second prompt-only iteration. Tagger version bumps 3→4; operator runs `engram tag --rerun --since 1970-01-01T00:00:00Z` to backfill.

### v3 dogfood data (the falsification)

23/23 thoughts at v3. Entities distributions broke into three buckets:

| pattern | count | examples |
|---|---|---|
| Clean (canonical names only) | 12/23 | `9b6e27d5` (Redis/PostgreSQL/Cassandra), `22bccb3a` (Cap'n Proto/Protocol Buffers), `5aacd2d8` (Hummingbird), `b67db532` (TCGplayer), `c5617a9e` (Rust/C/Go/Zig), `fe07320f` (Bazel/Make/Nix) |
| Mild padding (one or two off-pattern items) | 5/23 | `86c3392f` (... + "100MB", "1MB"), `96b470cf` (... + "reranker"), `1802ff2c` (... + "long-polling"), `ce83b7ba` (... + "ollama/qwen3-coder:30b v1") |
| Heavy padding (multi-word descriptive phrases) | 3/23 | `74eb781c`, `45cd2001`, **`047d0ce8`** |

The smoking gun was `047d0ce8`. Content: *"The agent memory protocol provides five operations: writing notes, querying them by similarity or recency, fetching by id, and marking notes untrusted. Querying combines embedding-based and lexical signals, optionally re-scored by a cross-encoder."* v3 entities: `["agent memory protocol", "embedding-based", "lexical signals", "cross-encoder"]`.

Four of seven verbatim items in the v3 prompt's "NOT entities" negative-example list appeared in the entities output. The model saw the explicit exclusion right next to those exact phrases — and emitted them anyway because they appeared as noun phrases in the thought's content. **The v3 negative-example list was counterproductive**: putting the forbidden phrases prominently in the system prompt next to a thought that uses them reinforced their salience as nameable things, rather than excluding them. The model's heuristic — "noun phrases in the content are entities" — is stronger than rule-language can override.

Three distinct failure shapes were operating:
1. **Anchor failure on proper-noun-free content** (`047d0ce8`) — the load-bearing one. When no proper nouns are present, the model substitutes descriptive noun phrases. The empty case needs to be the default framing, not a constraint.
2. **Slot-padding on proper-noun-anchored content** (`74eb781c`, `45cd2001`, `12a2626f`) — the model treated `maxItems: 5` as "fill 5 slots."
3. **Generic-concept padding** (`reranker`, `long-polling`, `MVCC`, `VACUUM`, `100MB`) — edge cases; the canonical/descriptive line is genuinely fuzzy.

Separately, scope-aware vocab began sacrificing topic precision in practice. The v2/v3 vocab-section sentence ("prefer the established form. Coin a new term only when the prose introduces something genuinely unseen") read as a hard preference and the model picked vocab terms even when more precise terms would have fit.

### v4 prompt changes

**1. Entities description restructured to lead with empty.** Replaced the v3 description (which led with "canonical proper names ... " and appended "RETURN [] IF ...") with:

> entities: default to []. Only add an item if the thought explicitly names a specific entity by its canonical name — a project, product, library, tool, technology, or organization that exists by that name outside this thought. Examples of valid entities: "engram", "pgvector", "PostgreSQL", "MCP", "TCGplayer", "Cap'n Proto", "Rust", "Hummingbird". Test before including any phrase: does this phrase NAME a specific thing that has its own canonical identity (yes → entity), or does the thought DESCRIBE an action or concept using a noun phrase (no → belongs in topics if anywhere)? Preserve the thought's casing, or use canonical casing if the thought is inconsistent. If you are unsure whether a phrase qualifies, omit it.

Two structural shifts: (a) empty is the default; the model has to justify adding rather than justify omitting; (b) the negative-example list is gone — replaced with a structural test ("NAME vs DESCRIBE") that generalizes across any content.

**2. `entities.maxItems` dropped 5→3.** Matches `topics`. Forces selectivity; reduces the "fill the slots" failure mode for proper-noun-anchored content.

**3. Vocab-section softening, from "dominate" to "tie-break":**

```diff
-When a concept in the thought matches one of these established terms,
-prefer the established form. Coin a new term only when the prose
-introduces something genuinely unseen.
+These are terms other thoughts in this scope have used. Use one when
+it accurately describes the thought's subject. Use a more specific or
+different term when no vocab term is a close fit — precision over
+consistency.
```

Vocab is now an option, not a constraint. The corpus-coherence benefit on broad subjects is preserved (vocab is still presented prominently) but precision wins ties.

Kind-isolation clause from v3 is retained unchanged.

### v4 expected behavior (hypotheses to verify post-deploy)

- `047d0ce8` re-tags with `entities: []` (or, at most, one canonical item if the model finds one).
- `74eb781c`, `45cd2001` lose their padding items and keep only canonical names.
- Topic precision improves on thoughts with distinctive content — more topic diversity, less reuse of high-frequency scope terms when a more specific term fits.
- No regression on the stably-classified controls (`5aacd2d8`, `b67db532`, the clean-bucket thoughts).
- The dominator → tie-breaker shift may modestly reduce cross-thought topic consistency on broad-subject thoughts — that's the deliberate trade.

### v4 dogfood plan

1. Bump local TOML `model_version = 4` (or remove the line so the bundled default applies).
2. `engram tag --rerun --since 1970-01-01T00:00:00Z` against the current corpus.
3. Re-pull entities arrays from the DB (same query shape as the v3 dogfood) and check the three failure-shape buckets shrink — especially `047d0ce8`'s heavy-padding case.
4. Re-run `kind_stability_diagnostic_with_vocab` to confirm vocab softening hasn't destabilized kind on bistable content beyond v3's pattern.
5. Capture findings as thoughts in `engram.m3.dogfood`.

### v4 lessons (worth recording independently)

- **Negative-example lists can be counterproductive.** Listing the phrases you want to forbid puts them in the system prompt context where the model treats them as relevant terms. Structural rules (apply-a-test) generalize better than enumeration (don't-do-these).
- **Lead with the default case.** v3's "list of names ... RETURN [] if none" framed the field's primary purpose as enumeration. v4's "default to [] ... only add if X" reverses the burden — the model has to justify including, not justify omitting.
- **`maxItems` is read as a fill target by the model.** Lowering matters more than the absolute value; the model expands to the cap.
- **Vocab framing matters.** "Prefer the established form" reads as a hard preference; "use when it accurately describes" reads as advisory. One-sentence wording changes affect the prior in measurable ways.

### v4 acknowledged limits

- Same kind cross-snapshot drift as v3 (22bccb3a). v4 doesn't claim to fix that — it's an inherent property of how model+vocab+corpus state interact at tag-time. Documented as advisory.
- Generic-concept padding edge cases (MVCC, VACUUM, long-polling) may or may not resolve under v4 — the structural test gives the model a clearer rule but humans disagree on the line too. Acceptable.
- If v4's lead-with-empty framing produces *too many* empty entities arrays (under-extraction), that's a new failure mode to watch for in dogfood. The structural test should mitigate this but it's a real risk.

v4 is the second of two M4.1 prompt iterations. If v4 dogfood confirms the failure modes are addressed without introducing new ones, M4.1 is done and the next milestone (M5 — selective relations) becomes the focus.
